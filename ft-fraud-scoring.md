# 📊 Методика комплаенс-скоринга для крупных P2P-сделок

Данная методика применяется сотрудниками департамента верификации для комплексного анализа пользователей, запрашивающих суточный лимит на P2P-сделки свыше **500 000 рублей**.

---

### 🧮 1. Таблица оценки факторов риска

Для расчета итогового скоринга используйте следующую шкалу баллов. Баллы по всем трем типам рисков суммируются.


| Тип риска | Анализируемый фактор | Индикатор / Признак | Баллы |
| :--- | :--- | :--- | :---: |
| **Цифровой** | Срок жизни аккаунта | Менее 7 дней с момента регистрации | **3** |
| | | От 7 до 30 дней с момента регистрации | **1** |
| | | Более 30 дней с момента регистрации | **0** |
| **Цифровой** | Геолокация и IP | Страна выпуска карты не совпадает со страной IP (без VPN) | **2** |
| | | Страна выпуска карты совпадает со страной IP-адреса | **0** |
| **Поведенческий** | Действия в интерфейсе | Скорость ввода данных быстрее ручной (копипаст реквизитов) | **2** |
| | | Зависание на этапе ввода SMS-кода более 3 минут | **2** |
| | | Хаотичные клики, частые обновления страницы | **1** |
| **Финансовый** | Маркеры дропперства | Наличие признаков психологического давления / работы по указке | **2** |

---

### ⚙️ 2. Регламент действий аналитика по результатам скоринга

После подсчета суммарного балла незамедлительно примените один из трех сценариев:

#### 🟢 0–3 балла — Низкий уровень риска
1. Выставите в системе статус риска: `Низкий`.
2. Одобрите лимит на P2P-сделки в автоматическом режиме.
3. Закройте внутреннюю задачу по верификации.

#### 🟡 4–6 баллов — Средний уровень риска
1. Выставите в системе статус риска: `Средний` и приостановите одобрение лимита.
2. Направьте пользователю запрос на дополнительную верификацию (KYC).
3. **Запросите:** Селфи пользователя с раскрытым паспортом в руках и физическим листом бумаги, где от руки написана сегодняшняя дата.
4. Проверьте подлинность фото после получения и примите финальное решение.

#### 🔴 7 и более баллов — Критический уровень риска
1. Выставите в системе статус риска: `Критический`.
2. Заблокируйте аккаунт пользователя на платформе до выяснения обстоятельств.
3. Зарегистрируйте инцидент безопасности в *Jira* по форме `RISK-BOARD`.
4. Поставьте в задаче *Jira* тег дежурного офицера безопасности для передачи кейса в работу.
